1. Headline verdict — your nested hypothesis is half wrong

Your instinct that "nesting" is a V2 story is partly right, but the guide (and your mental model) conflates two different things. Disentangling them is the single most important correction:

Category 1 — inlined nested structs WRONG to call V2

A store struct embedded by value inside the object, reached by a dot-path like {stats.strength} or {a.b.c}.

This already worked in V1. The Move Book documents it verbatim: "The path is a dot-separated list of field names, starting from the root object in case of nested fields," with the worked V1 example {metadata.description}.

Category 2 — references & dynamic fields genuinely V2

Following an ID reference or a dynamic field from your object to a separate on-chain object, and reading its fields — plus traversing collections.

This is the real V2 addition. The launch blog: V2 templates "can now reach beyond the object itself… access dynamic fields attached to an object and load related objects," and in V1 builders "couldn't reference collections, dynamic fields, or related objects inside their templates."

Bottom line: your guide's "Advanced: Richer Templates → Nested field access" bullet (the {bar.val}/{bar.baz.val} examples) describes a V1 capability and must not be sold as new in V2. The truly-new bullets in that same section — dynamic field references and collection traversal — are correct as V2 additions.

Boundary, sharpened (tested both standards on localnet): the line is not "struct vs object" — it's by value vs by reference. A wrapped object — a key, store struct (a first-class object with its own UID) embedded by value inside another object — is just inlined bytes, so its fields and its {inner.id} resolve under V1 and V2, identically (verified: {inner.val}→"777", {inner.id}→the wrapped UID). What V2 adds is following a reference / dynamic field to a separately-stored object — one that is not embedded by value. (Note: a key-only struct can't even be a field — a field type needs store — so the embeddable case is always key, store.)

3. You asked: if it was in V1, did the behavior change in V2?

For the inlined {a.b.c} case specifically: no documented behavior change. Both versions resolve a template by string-interpolating a dot-path over the object's fields; the resolution happens off-chain (indexer / RPC / GraphQL), not in the Move module, in both V1 and V2. V2's engine is a superset: same core interpolation, but a "path" may now also cross into dynamic fields, related objects, and collections, plus the guide-claimed :json and default-value forms.

What I could confirm about V2's resolution behavior, live:

  • Non-String leaves stringify — a u64 field rendered as "99" in the output.
  • Multi-token + literal composition in one property works ("STR {stats.strength} / DEF {stats.defense}").
  • On full success, display.error is null; a template referencing an unresolvable token populates display.error instead (the V2 RPC returns a { data, error } shape).
Resolved empirically (probe package minted on localnet, 18+ templates set, resolved display read back — 2026-06-29):
  • Depth: {nested.l2.l3.deep} (4 levels) → "1234". No depth limit observed.
  • Leaf stringification: bool→"true"/"false"; address & {id}→full 0x+64-hex; u128→full decimal; vector<u8>→decoded as text ("ABC"); Option::some(v)→inner value.
  • Not directly renderable: bare vector<u64> and bare structs → error ("unexpected vector/struct"); use :json.
  • Transforms {x:T} — full allowlist (from the resolver's own parser error): base64 · bcs · hex · json · str · ts · url. :json serializes structs/vectors to JSON (ints become JSON strings). No :default → the guide's "default values" feature does not exist.
  • Index access {vec[i]}: not supported (parse error). "Collection traversal" is via :json on the whole collection, not element indexing.
  • Missing field ({nope}) → key present in data with value null. Invalid expression → key omitted from data + message appended to display.error ({ code:"displayError", error:"…" }). Two distinct behaviours.
  • V1 vs V2 resolver deltas (ran the same type under both): V1 has no transforms (any :json → "Failed to parse format"); and in V1 one bad format string blanks the entire display.data (returns empty + error), whereas V2 isolates the failure to the offending key. V2 is a strictly better resolver.
Still open: the exact template syntax for following a reference / dynamic field to a separate object (the headline V2 feature) — not reproduced blind; needs the V2 template-syntax doc.

5. Concrete fixes for display-v2-guide.md

  1. "Advanced: Richer Templates → Nested field access" — relabel. Move the {bar.val} / {bar.baz.val} dotted-path examples out of the "new in V2" framing; note explicitly that V1 already supported inlined nested-struct dot-paths. Keep dynamic field references and collection traversal as the genuine V2 nesting story (following references to separate objects).
  2. Architecture → "stored in a Bag keyed by TypeName" — reword to a derived-object model: the Display<T> lives at an address derived from the registry + a DisplayKey<T>, which is what makes the ID deterministic.
  3. "Permit pattern is the preferred long-term approach" — soften to "an alternative creation path that proves type ownership through the module system without a Publisher." Don't assert it's preferred.
  4. "~4,500 existing V1 Display objects" and "second migration sweep" — either source them or hedge ("a few thousand", "a planned follow-up sweep"). Neither figure is in the primary sources.
  5. RPC table — keep the JSON-RPC row (sourced). Add a hedge to the GraphQL/gRPC rows ("per the launch materials; exact schema not independently verified here").
  6. Timeline — the dates are confirmed as announced; add a one-line caveat that the July 31 2026 cutoff is a forward-looking deprecation that could move.
  7. V1 event nuance (optional, strengthens the "Why V2" case) — in V1, field add/edit/remove do not bump the version or emit an event (a TODO in the framework source), so even V1's event-discovery story was incomplete. Good ammunition for problem (c).

6. Omissions worth adding

  • The exact V1 vs V2 struct definitions, side by side — readers grok the delta instantly: Display<phantom T: key> { id, fields, version: u16 } (V1, owned, self-versioned) vs Display<phantom T> { id, fields, cap_id: Option<ID> } + separate DisplayCap<T> (V2, shared + cap).
  • Off-chain resolution. Make explicit that template resolution happens off-chain (indexer / RPC / GraphQL), not in the Move module — in both versions. This is why the "richer engine" is a client-side change, and why behavioral edge-cases (§3) aren't in the Move source.
  • The V2 { data, error } response shape — V2 RPC returns resolved fields in data and per-field failures in error (your playground already surfaces this). A nice contrast with V1.
  • The cap-gating security upgrade — your guide hints at it; state it plainly: V1's Display has store and was mutated by whoever held it (a shared/owned foot-gun); V2 splits read (shared Display) from write (owned DisplayCap).
  • Open items for a follow-up pass (see §7) — the :json/defaults syntax, GraphQL/gRPC specifics, and nesting depth/missing-field semantics are genuinely unconfirmed; flag rather than assert.
